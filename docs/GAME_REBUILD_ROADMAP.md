# SwanSong Originals rebuild roadmap

The anthology is being rebuilt as ten production games, not decorated as ten
larger versions of the same microgame. The reusable system is proven on one
title, measured with people, extracted into the SDK, and only then applied to
the next title.

The release contract is [GAME_QUALITY_STANDARD.md](GAME_QUALITY_STANDARD.md).
The current collection remains in descriptive report mode:

```sh
make quality
```

One Last Lap is the intended first strict canary. It is deliberately **not**
enabled as a passing strict gate yet. This command is expected to fail until
its production rebuild and evidence are complete:

```sh
make quality-strict GAME=one-last-lap
```

## Baseline problem

The current games compile, reset deterministically, reach documented outcomes,
and have native graphical playfields. Those are valuable foundations. They do
not yet meet the production standard:

- openings automatically leave the intro after 36 frames or any input;
- nine games have no manifest music asset;
- no game declares an SDK SFX asset;
- every cartridge declares no save payload;
- every game declares zero sprites and source inspection finds no second-layer
  presentation path;
- resource budgets repeat hardware maxima instead of measured ceilings;
- deterministic success routes are all under one minute;
- retained native captures show gameplay, not separately reviewed title and
  result presentations; and
- there is no identifier-free, uncoached human-playtest evidence.

The exact per-game inventory is generated by the quality audit rather than
maintained by hand.

## Stage 0 — unblock the production runtime

Do this before adding background music to every game:

1. Correct SFX arbitration to prefer a silent voice, respect an optional
   effect channel, steal only the lowest-priority eligible voice, and restore
   the resolved held music state.
2. Make instrument attack/release affect output, or remove the false authoring
   promise until it does.
3. Add deterministic music ducking and explicit title/gameplay/result cue
   transitions.
4. Provide reusable fixed-capacity title/menu, pause, transition, HUD, result,
   and small particle/feedback primitives.
5. Add SwanSong cycle, missed-VBlank, audio-channel, and sprite-overflow
   measurements needed by the strict five-minute profile.
6. Add `swan quality`, `swan audio lint`, and `swan feel` equivalents to the
   public SDK after the anthology-side contract proves their requirements.
7. Extend scenario-outcome reset contracts with an explicit count or lower
   bound. A fresh gameplay boot already records a session reset, so the current
   `required` spelling cannot by itself prove that a result-screen replay added
   another reset; until the schema grows, pair the final post-reset state with
   the failure/result audio marker that proves the preceding run occurred.

Until those changes land, one title can prototype presentation with current
SDK scenes and graphics, but it cannot honestly pass the complete strict
runtime/audio profile.

## Stage 1 — One Last Lap production canary

One Last Lap is first because it exercises continuous action, rivals, hazards,
multiple endings, timing, replay value, and dense audiovisual feedback without
requiring a large content engine.

### Core play rebuild

- Replace the single progress counter presentation with three authored course
  sections: opening straight, technical middle, and rescue/final sprint.
- Give rivals visible lane intent and collision/overtake behavior that creates
  readable decisions rather than decorative traffic.
- Turn battery into a risk/reward system: efficient drafting and clean braking
  preserve charge; collisions, conflicting input, and maximum acceleration
  waste it.
- Keep the tow decision, but telegraph the rescue window and make helping cost
  time or battery so both endings express a real choice.
- Target a 3–6 minute first Standard run, with Assist and Challenge changing
  named timing/pressure values rather than hidden adaptation.
- Save best time per difficulty, clean-lap count, cooperative/solo endings,
  tutorial completion, and audio/difficulty options.

### Presentation rebuild

- Persistent illustrated title with authored pixel lettering and title music.
- Deterministic attract lap after 8–12 seconds idle.
- One-lap playable tutorial: steer, accelerate, brake/draft, then tow.
- Second background for road/HUD depth and sprites for racers, hazards, tow
  marker, particles, and result celebration.
- Pause, controls, restart confirmation, distinct cooperative/solo/failure
  results, best-time comparison, and two-action retry.
- Gameplay track with A/B/turnaround form, title loop, and prioritized UI,
  engine, pass, collision, low-battery, tow, lap, win, and loss effects.

### Canary exit gate

One Last Lap leaves this stage only when all of the following are true:

- existing portable-model, ROM, and SwanSong tests still pass;
- `make quality-strict GAME=one-last-lap` reports zero blockers;
- a five-minute profile meets CPU/RAM/VRAM/sprite/audio limits;
- title, gameplay, and each meaningful result are inspected at native size;
- five uncoached local players meet the first-run thresholds; and
- recurring implementation work is extracted into an SDK API, recipe,
  asset rule, documentation item, or regression test before title two starts.

Do not add One Last Lap to an aggregate strict list before that exit gate is
real.

## Stage 2 — prove the arcade production kit

### Orbital Courier

- Build a sequence of compact contracts with different parcel weight, route,
  hazard, and fuel tradeoffs rather than one solved grid.
- Add moving traffic, route previews, optional risky shortcuts, delivery rank,
  and saved best rank/time per contract.
- Use a world layer plus HUD/route overlay, a sprite courier/parcel/traffic
  set, title/dispatch/results music states, and clear pickup/blocked/fuel/deliver
  feedback.
- Preserve deterministic route validation and use it to verify every authored
  contract remains solvable.

### Harpoon Moon

- Expand the hunt into readable target behaviors, lure windows, charged-shot
  timing, and a multi-phase leviathan encounter.
- Make oxygen pressure legible without constant text and give missed shots a
  recoverable cost rather than dead time.
- Save hunt ranks, best oxygen remainder, discovered creatures, and difficulty.
- Use sprite animation, parallax water/sky, channel-4 noise percussion, charge
  pitch, impact effects, and distinct boss/result music.

The arcade kit is stable only when these two games need no private framework
changes beyond game-owned rules and assets.

## Stage 3 — prove puzzle, strategy, and vertical kits

### Bug Witch

- Organize puzzles into short spellbooks with escalating concepts, optional
  hints, undo, clear feedback about why a circuit failed, and saved completion
  medals.
- Teach one familiar at a time through a playable first puzzle.
- Animate signal flow on the second layer and use sprites for familiars,
  selection, sparks, and successful output.

### Turncoat Tactics

- Add several small battle maps, explicit movement/attack/recruit previews,
  enemy intent, meaningful unit differences, and a short campaign result.
- Ensure recruiting creates a tactical tradeoff rather than a strictly better
  attack.
- Save cleared maps and best turn/rank. Profile worst-case unit overlap against
  the 24-sprites-per-scanline production ceiling.

### Rotate Dungeon

- Make this the real vertical-orientation proof: 144×224 authoring, normalized
  native evidence, Y-cluster primary navigation, and X-cluster alternative.
- Animate the room transformation and preserve player comprehension across
  both geometry states.
- Add themed room sets, optional route hint, room records, and a final escape
  sequence instead of five near-identical checks.

This stage extracts grid focus, path preview, undo/hint, vertical composition,
and scanline-aware unit helpers into the recipes.

## Stage 4 — prove observation and diagnostic play

### Radio Ghost

- Make audio direction and tuning behavior part of play: stereo position,
  signal texture, gain/noise tradeoffs, and clue assembly should agree.
- Add several nightly signal sets and saved discoveries/best remaining time.
- Provide a visual alternative for every required audio cue while retaining
  the sound-first identity.

### Pocket Kaiju Observatory

- Give the subject multiple readable behavior cycles and environmental clues,
  not only three timed evidence bits.
- Add a small saved field album, photo ranks, replayable day conditions, and a
  tutorial that teaches distance, disturbance, hiding, and zoom in context.
- Separate subject/camera sprites from scenery and use the overlay layer for
  focus, disturbance, and photographic feedback.

### Scrapframe Garage

- Turn each repair into observe → diagnose → choose part → install/test, with
  symptoms that teach a reusable rule rather than a one-step match.
- Add several authored customer sets, optional diagnostic help, accuracy/time
  rank, and saved best shifts.
- Make tool selection, wrong-part rejection, successful repair, and shift
  results audiovisual events rather than icon swaps.

This stage extracts stereo/visual cue equivalence, collection/album saves,
diagnostic-step UI, and non-action difficulty controls.

## Stage 5 — rebuild Mote Sound Terminal as the audio showcase

Mote Sound Terminal should validate the mature audio system rather than freeze
the limitations of the first driver.

- Persistent title, track browser, Now Playing, help, options, and deterministic
  attract visualization.
- At least three genuinely developed 45–90 second four-voice compositions with
  recognizable hooks, contrasting form, and composed loop seams.
- Per-channel mute/solo, scope modes, pause/resume, restart, volume, and saved
  favorites/last track without losing deterministic phase semantics.
- Speaker-safe mono plus restrained stereo, intentional channel-3 sweep and
  channel-4 noise, and SwanSong WAV evidence across every loop.
- A stable utility outcome screen and replay contract instead of forcing a
  conventional win/loss.

The audio showcase passes only when a listener test covers hardware-chip,
game-soundtrack, and borrowed-genre listeners and the intended lane prefers the
revision without third-loop fatigue dominating.

## The learning loop after every game

Before work begins on the next title, record:

1. What code was repeated?
2. What asset conversion or evidence step required manual repair?
3. Which budget was closest to its ceiling?
4. Which automated check passed while a person was still confused?
5. Which feedback or music decision improved or hurt replay desire?

Every recurring item becomes exactly one of:

- a public `swan_` runtime API;
- a manifest/asset validation rule;
- a recipe improvement;
- a SwanSong evidence/profiling feature;
- a Desktop Studio control; or
- a regression test/documentation item.

Game-owned rules remain in portable model modules. Art direction, character
identity, pacing values, level content, and musical themes do not move into the
framework.

## Collection release gate

The rebuilt collection is releasable only when:

- all ten games pass existing tests and their own strict quality audit;
- every game has persistent title/menu/tutorial/pause/result or declared
  utility-outcome flow;
- every game has inspected background music and SFX evidence;
- all saves survive the SDK corruption/interruption matrix;
- every runtime profile has five-minute CPU, VBlank, RAM, tile, palette, sprite,
  and scanline headroom;
- every game has identifier-free human-playtest evidence;
- physical WonderSwan Color/SwanCrystal display and speaker checks are reported
  separately from SwanSong evidence; and
- no Mednafen or alternate emulator appears in build, testing, or acceptance.

The collection does not receive a production claim because ten ROMs boot. It
earns that claim when ten distinct games teach, respond, finish, save, replay,
and survive both machine and human review.
